## 🏗️ Backend Architecture Ideas

> **Note**: These are design concepts for future backend implementation. Currently UI-only mode.

### 📁 Hierarchical Storage Structure

When backend storage (Supabase) is connected, the system will use a subject-based bucket hierarchy:
├── 📁 HRM (Subject Folder - permanent)
│   ├── 📁 HR_CaseStudy_2026-03-01 (Assignment Folder)
│   │   ├── 📄 student1_username_file.pdf
│   │   └── 📄 student2_username_file.docx
│   └── 📁 Lab_Report_2026-03-15 (Another Assignment)
│       └── 📄 student submissions...
├── 📁 CS101 (Subject Folder - permanent)
│   └── 📁 Assignment_Title_Date
│       └── 📄 submissions...
└── 📁 NewSubject
    └── ...
```

### 🎯 Design Principles

| Principle | Description |
|-----------|-------------|
| **Subject-First** | Each subject has one permanent folder |
| **Assignment Subfolders** | New assignment = new subfolder with title+date |
| **Organized Storage** | Easy to browse by subject → assignment → submissions |
| **No Duplicates** | Subject folders persist, only assignment folders change |

### 🔮 Future Backend Integration

- **Subjects**: Permanent folders created once
- **Assignments**: Subfolders created dynamically
- **Submissions**: Files organized by assignment
- **Benefits**: Scalable, organized, human-readable structure

## 🗄️ Database Schema Design

> **Approach**: Starting with core domain tables (`subjects`, `assignments`, `submissions`) and a `profiles` table for Role-Based Access Control (RBAC).

### Table: `profiles` (Closed Registration & RBAC)

The portal uses a **Closed Registration System** (Sign-ups are disabled). Admins create accounts manually in Supabase. A database trigger automatically creates a row here when a new user is added.

| Column | Type | Description |
|--------|------|-------------|
| `id` | UUID PRIMARY KEY | References Supabase `auth.users.id` |
| `email` | VARCHAR(255) | User's email address |
| `name` | VARCHAR(100) (nullable) | Student/Admin display name |
| `role` | VARCHAR(20) DEFAULT 'student' | Determines access ('admin' or 'student') |
| `created_at` | TIMESTAMP | Auto-creation timestamp |

### Table 1: `subjects`

Permanent subject folders in storage. Created once, persist forever.

| Column | Type | Description |
|--------|------|-------------|
| `id` | VARCHAR(10) PRIMARY KEY | Subject code (HRM, CS101, etc.) - also folder name |
| `name` | VARCHAR(100) | Full subject name ("Human Resource Management") |
| `description` | TEXT (nullable) | Optional subject description |
| `color` | VARCHAR(7) (nullable) | Hex color code for UI (#8b5cf6) |
| `folder_id` | VARCHAR(255) (nullable) | External storage reference (Supabase bucket/path) |
| `created_at` | TIMESTAMP | When subject was created |
| `updated_at` | TIMESTAMP | Last modification time |
| `is_active` | BOOLEAN DEFAULT true | Soft delete flag |

**Example Records:**
```
| id    | name                      | color   | created_at          |
|-------|---------------------------|---------|---------------------|
| HRM   | Human Resource Management | #10b981 | 2026-03-01 10:00:00 |
| CS101 | Data Structures           | #8b5cf6 | 2026-03-01 10:30:00 |
```

### Table 2: `assignments`

Assignments belong to subjects. Each assignment creates a subfolder.

| Column | Type | Description |
|--------|------|-------------|
| `id` | UUID PRIMARY KEY | Unique assignment identifier |
| `subject_id` | VARCHAR(10) FOREIGN KEY | References `subjects.id` - which subject this belongs to |
| `title` | VARCHAR(200) | Assignment title |
| `description` | TEXT (nullable) | Assignment instructions/details |
| `assign_date` | DATE | When assignment was given |
| `due_date` | DATE | Assignment deadline |
| `status` | ENUM/VARCHAR | active / inactive / expired |
| `folder_name` | VARCHAR(255) | Generated folder name (Title_Date format) |
| `folder_id` | VARCHAR(255) (nullable) | External storage reference (Supabase path) |
| `created_at` | TIMESTAMP | When created |
| `updated_at` | TIMESTAMP | Last modified |
| `created_by` | VARCHAR(100) | Admin who created it |

**Status Logic:**
- `active` - Assignment accepting submissions
- `inactive` - Manually disabled by admin
- `expired` - Due date passed (can be auto-calculated or stored)

**Example Records:**
```
| id | subject_id | title                  | assign_date | due_date   | status  | folder_name                      |
|----|------------|------------------------|-------------|------------|---------|----------------------------------|
| 1  | HRM        | HR Management CaseStudy| 2026-03-01  | 2026-03-15 | active  | HR_Management_CaseStudy_2026-03-01|
| 2  | HRM        | Lab Report 2           | 2026-03-10  | 2026-03-25 | active  | Lab_Report_2_2026-03-10         |
| 3  | CS101      | Data Structures Quiz   | 2026-03-05  | 2026-03-12 | expired | Data_Structures_Quiz_2026-03-05  |
```

├─────────────────┤       ├─────────────────────┤
│ PK: id          │──────<│ FK: subject_id      │
│     name        │   1:M │     title           │

│     is_active   │       │     due_date        │

         │                          │

    (permanent)           (one per assignment)

### Storage Integration Flow

**When Subject Created:**
1. Insert into `subjects` table
2. Create permanent folder in Supabase Storage using the subject ID.
3. Store path reference back to database

**When Assignment Created:**
1. Insert into `assignments` table with `subject_id` FK
2. Create subfolder inside subject's bucket path
3. Store path reference in `folder_id`
4. Folder name format: `{sanitized_title}_{assign_date}`

**Query Examples:**

```sql
-- Get all active subjects with assignment count
SELECT s.id, s.name, s.color, COUNT(a.id) as assignment_count
FROM subjects s
LEFT JOIN assignments a ON s.id = a.subject_id AND a.status = 'active'
WHERE s.is_active = true
GROUP BY s.id;

-- Get all assignments for a subject
SELECT * FROM assignments 
WHERE subject_id = 'HRM' 
ORDER BY assign_date DESC;

-- Get assignments with status
SELECT a.*, s.name as subject_name, s.color
FROM assignments a
JOIN subjects s ON a.subject_id = s.id
WHERE a.status = 'active' AND a.due_date >= CURRENT_DATE;

-- Get all submissions for an assignment
SELECT sub.*, a.title as assignment_title, s.name as subject_name
FROM submissions sub
JOIN assignments a ON sub.assignment_id = a.id
JOIN subjects s ON a.subject_id = s.id
WHERE sub.assignment_id = 'assignment-uuid';

-- Get all submissions by a student
SELECT sub.*, a.title as assignment_title, s.name as subject_name
FROM submissions sub
JOIN assignments a ON sub.assignment_id = a.id
JOIN subjects s ON a.subject_id = s.id
WHERE sub.student_id = 'student-123';
```

### File Renaming Workflow (Frontend Logic)

**Important**: File renaming happens on the **frontend** before sending to backend. This keeps the backend simple and clean.

**Process:**
1. Student selects file to upload
2. **Frontend** renames file: `{student_name}_{original_filename}`
3. Frontend sends renamed file to backend
4. Backend simply stores the file as-is (no renaming logic needed)
5. Backend saves submission record with metadata

**Example:**
```
Original:  hrm 1st task.pdf
Student:   Ali
Rename:    Ali_hrm 1st task.pdf  ← Frontend does this
Upload:    Ali_hrm 1st task.pdf  → Backend receives already-renamed file
Store:     Subject/Assignment/Ali_hrm_1st_task.pdf
```

**Frontend Logic:**
```javascript
// Before uploading, rename the file
const originalFile = event.target.files[0];
const studentName = 'Ali'; // from logged in user
const newFileName = `${studentName}_${originalFile.name}`;

// Create new File object with renamed name
const renamedFile = new File([originalFile], newFileName, {
  type: originalFile.type,
  lastModified: originalFile.lastModified,
});

// Upload renamedFile to backend
uploadToBackend(renamedFile);
```

**Benefits:**
- ✅ Backend stays simple (no file manipulation)
- ✅ Frontend controls naming convention
- ✅ Easy to debug and modify
- ✅ No filename collisions between students
- ✅ Human-readable naming convention

**Backend Receives:**
- File already renamed (`Ali_hrm 1st task.pdf`)
- Student info (name, id)
- Assignment ID
- Backend just stores file and saves record

### Submission Workflow (Complete Flow)

## 🏗️ Core Application Flow & Logic (Backend Developer Guide)

This section outlines the exact step-by-step logic for how the application operates. Any new developer should read this to understand the data flow between the UI, Database, and Storage.

### 1. Subject Management (Admin Only)
- **Action**: Admin clicks "Manage Subjects" and creates a new subject.
- **Database**: A new record is saved in the `subjects` table.
- **Storage**: A main, permanent folder is created in the Supabase Storage bucket using the subject ID (e.g., `HRM` or `CS101`).
- **UI Impact**: This subject now appears globally:
  - In the Student's "My Subjects" view.
  - In the Admin's "Create Assignment" dropdown.

### 2. Assignment Creation (Admin Only)
- **Action**: Admin selects an active subject (e.g., `HRM`) and generates a new assignment.
- **Database**: A new record is created in the `assignments` table, linked to the `subjects` table.
- **Storage**: A sub-folder is created inside the selected subject's main folder.
  - Format: `SUBJECT/Assignment_Title_Date/`
  - Example 1: `HRM/Physical_resources_3182023/`
  - Example 2: `HRM/Internal_resources_3182023/`
- **Rule**: Every topic/assignment has its own isolated sub-folder. Files across different assignments never mix. No matter how many subjects exist (e.g., 5-10+), they strictly follow this hierarchical structure.

### 3. Assignment Submission (Student)
- **Action**: Student uploads an assignment file. 
- **Database**: A record is created in the `submissions` table.
- **Storage**: The renamed file (`StudentName_OriginalName`) is saved in the specific assignment's sub-folder.
- **UI Impact**: 
  - **For Student**: The "My Submissions" progress UI updates to reflect what has been submitted and what is still pending.
  - **For Admin**: The Admin dashboard updates to show exactly which students have submitted the assignment and who is left.

### 4. Problem Resolution / Re-submission Workflow (Current Scope)
- **Current Constraints**: The Student UI is currently built strictly for **Uploading** and viewing **History**. Students do **NOT** have access to "Delete" or "Resubmit" buttons.
- **Manual Correction Process**: 
  - If a student makes a mistake (e.g., uploads the wrong file format or wrong document), they must inform the Admin.
  - The Admin has to manually log into the **Supabase Dashboard**.
  - **Step A**: The Admin deletes the incorrect file from Supabase Storage.
  - **Step B**: The Admin deletes the corresponding submission record from the `submissions` table.
  - Once deleted, the UI will automatically show the assignment as "Pending" for the student again, allowing them to re-upload.
- **Future Roadmap**: We might add self-serve resubmission features for students in the future, but it is explicitly **out of scope** for the current phase.

## 🔐 API & Security Architecture (Method 1: In-App Admin + RLS)

> **Decision**: We chose **Method 1 — In-App Admin Dashboard** as the way the Admin will manage Subjects, Assignments, and view Submissions. This section explains what it is, why we chose it, and exactly how security works.

### 🤔 What Is Method 1?

The Admin manages everything (create subjects, create assignments, view submissions) directly through the **AssignPortal web app itself** — using the same React frontend that students use, but with extra admin-only screens and buttons visible only to admin users. There is **no need** for the Admin to go to the Supabase dashboard for day-to-day operations.

### ✅ Why We Chose This Method

| Reason | Explanation |
|--------|-------------|
| **Already Designed** | Admin Dashboard UI screens (Manage Subjects, Create Assignment, View Submissions) are already designed in the frontend |
| **User Friendly** | Admin fills a form, clicks a button — done. No need to learn Supabase dashboard |
| **Professional** | Single app for both Admin and Student. Role determines what they see |
| **Secure** | Security is enforced at the **database level** by Supabase RLS, not by hiding buttons |
| **No Secret Keys in Frontend** | Only the safe, public `anon` key is used. The real security is in RLS policies |

### 🔑 Supabase Connection (Frontend Setup)

When you create a Supabase project, it gives you **2 things** to connect your app:

| Item | Example | Where to Use | Safe to Expose? |
|------|---------|--------------|-----------------|
| **Project URL** | `https://abcxyz.supabase.co` | `.env` file in React app | ✅ Yes |
| **Anon (Public) Key** | `eyJhbGciOiJIUzI1NiIs...` (long token) | `.env` file in React app | ✅ Yes |

```env
# .env file in React project
VITE_SUPABASE_URL=https://abcxyz.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIs...
```

**⚠️ Important**: The `anon` key is **NOT a secret**. It is designed to be in the frontend. It only lets you *reach* the Supabase API — it does **NOT** give permission to read/write data. Actual data permissions are controlled entirely by **RLS Policies** (explained below).

### 👥 User Roles in the System

There are exactly **2 roles** in AssignPortal:

| Role | How Account is Created | What They Can Do |
|------|------------------------|------------------|
| **Admin** | Created manually in Supabase Auth. Role set to `admin` in `profiles` table | Create/Edit/Delete Subjects & Assignments. View all student submissions. Delete incorrect submissions from Supabase dashboard |
| **Student** | Created manually by Admin in Supabase Auth. Role auto-set to `student` (default) via DB trigger | View assigned subjects. Upload assignment files. View own submission history |

**Role Check Flow (Every Page Load / API Call):**
```
User Opens App
  → Supabase checks: Is user logged in? (JWT Token)
  → App queries: SELECT role FROM profiles WHERE id = auth.uid()
  → If role = 'admin'  → Show Admin Dashboard, Admin menus
  → If role = 'student' → Show Student Dashboard, Upload page
  → If not logged in    → Redirect to Login page
```

### 🛡️ RLS (Row Level Security) — The Real Security

RLS is the **core security mechanism**. Even if someone steals the Anon Key or tries to call the API directly (bypassing the UI), the database itself will **block unauthorized actions**. Below are the exact policies we will implement:

#### Table: `profiles`

| Policy Name | Action | Rule | Description |
|-------------|--------|------|-------------|
| `Users can view own profile` | SELECT | `auth.uid() = id` | Users can only read their own profile row |
| `No public insert` | INSERT | `false` | Nobody can insert — only the DB trigger creates profiles |
| `No public update` | UPDATE | `false` | Role changes are done by Admin directly in Supabase dashboard |

#### Table: `subjects`

| Policy Name | Action | Rule | Description |
|-------------|--------|------|-------------|
| `Everyone can view active subjects` | SELECT | `is_active = true` | Both Admin and Students can see active subjects |
| `Only admin can create subjects` | INSERT | `(SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'` | Only admin role can add new subjects |
| `Only admin can update subjects` | UPDATE | `(SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'` | Only admin can edit subject details |
| `Only admin can delete subjects` | DELETE | `(SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'` | Only admin can remove subjects |

#### Table: `assignments`

| Policy Name | Action | Rule | Description |
|-------------|--------|------|-------------|
| `Everyone can view assignments` | SELECT | `true` | All logged-in users can see assignments (students need to see their pending work) |
| `Only admin can create assignments` | INSERT | `(SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'` | Only admin can create new assignments |
| `Only admin can update assignments` | UPDATE | `(SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'` | Only admin can edit assignment details or change status |
| `Only admin can delete assignments` | DELETE | `(SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'` | Only admin can remove assignments |

#### Table: `submissions`

| Policy Name | Action | Rule | Description |
|-------------|--------|------|-------------|
| `Students see own submissions` | SELECT | `auth.uid() = student_id` | Students can only view their own submission records |
| `Admin sees all submissions` | SELECT | `(SELECT role FROM profiles WHERE id = auth.uid()) = 'admin'` | Admin can view every student's submissions |
| `Students can submit` | INSERT | `auth.uid() = student_id` | Students can create submission records (for their own ID only) |
| `No student update/delete` | UPDATE/DELETE | `false` (for students) | Students cannot edit or delete submissions. Admin handles corrections manually via Supabase dashboard |

#### Storage Bucket: `assignments` (File Storage)

| Policy Name | Action | Rule | Description |
|-------------|--------|------|-------------|
| `Students can upload files` | INSERT | Authenticated users can upload to their designated assignment folder | Students can upload assignment files |
| `Admin can read all files` | SELECT | Admin role can access all folders and files | Admin can download/view any student's submission |
| `Students can read own files` | SELECT | Student can only access files they uploaded | Students can only see their own uploaded files |
| `Only admin can delete files` | DELETE | Admin role only | Only admin can remove incorrect files from storage |

### 🔒 Security Model Summary

```
┌─────────────────────────────────────────────────────────────────┐
│                    SECURITY LAYERS                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Layer 1: Authentication (Supabase Auth)                        │
│  ├── Closed Registration (no public sign-ups)                   │
│  ├── Admin creates all accounts manually                        │
│  └── JWT token issued on login                                  │
│                                                                 │
│  Layer 2: Role Check (profiles table)                           │
│  ├── role = 'admin'   → Full access to manage portal            │
│  └── role = 'student' → Limited to upload & view own data       │
│                                                                 │
│  Layer 3: RLS Policies (Database Level)                         │
│  ├── Even if someone bypasses the UI, database blocks them      │
│  ├── Students CANNOT create subjects/assignments                │
│  ├── Students CANNOT see other students' submissions            │
│  └── Students CANNOT delete any data                            │
│                                                                 │
│  Layer 4: Storage Policies                                      │
│  ├── File access controlled per-user                            │
│  └── Only admin can delete files                                │
│                                                                 │
│  Public Anon Key = Safe to expose (just a "door key")           │
│  RLS Policies    = The real "lock" (database decides who enters)│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 🚫 What Happens If Someone Tries to Hack?

| Attack Scenario | What Happens | Why |
|-----------------|--------------|-----|
| Student tries to create a subject via API | ❌ **Blocked** | RLS policy checks role, finds 'student', rejects INSERT |
| Someone finds the Anon Key and calls API | ❌ **Blocked** | Without a valid login JWT, Supabase rejects all requests |
| Student tries to view another student's submissions | ❌ **Blocked** | RLS policy: `auth.uid() = student_id` — only own data visible |
| Student tries to delete their submission | ❌ **Blocked** | DELETE policy is `false` for students |
| Admin creates subject via the app | ✅ **Allowed** | RLS checks role, finds 'admin', allows INSERT |

**Simple Rule**:

### 🔐 Types of Keys for AI Developers
